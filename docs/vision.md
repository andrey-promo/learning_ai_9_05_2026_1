# Техническое видение

Отправная точка и технический проект для проверки идеи из [idea.md](idea.md).

---

## 1. Технологии

Целевая платформа разработки: **macOS** (Windows не учитываем).

| Область | Выбор | Примечание |
|--------|--------|------------|
| Язык | **Python 3.12** | Одна зафиксированная версия для проекта |
| Зависимости | **uv** | `pyproject.toml` и lock-файл; установка и запуск через `uv sync`, `uv run`; среди зависимостей — **`python-dotenv`** для локального `.env` |
| LLM | Пакет **openai** (официальный клиент) | Доступ к моделям через **OpenRouter**: `base_url` и ключ из переменных окружения |
| Telegram | **aiogram 3.x**, режим **long polling** | Без webhook и без отдельного HTTP-сервера для приёма апдейтов |
| Локальный запуск в контейнере | **Docker** | Удобный локальный старт; тот же образ используется при деплое |
| Облачный деплой | **Railway** | Деплой Docker-образа; переменные окружения задаются через Railway Dashboard |
| Сборка и команды | **GNU Make** | Цели для установки зависимостей, запуска **без** Docker и **с** Docker |

**Сборка:** для этого проекта нет отдельного компилируемого артефакта — «сборка» означает синхронизацию зависимостей (`uv`) и при необходимости сборку Docker-образа. Точка входа — обычный запуск Python-процесса бота.

---

## 2. Принципы разработки

- **KISS:** минимум сущностей и файлов; цель v1 — текстовый ответ бота через LLM; без очередей, шин событий и отдельного публичного API, пока это не нужно для проверки идеи.
- **ООП:** классы там, где они уменьшают связность и проще сопровождать код; не вводим классы «ради паттерна».
- **Один класс — один файл**; имя модуля в `snake_case`, публичный класс — в `PascalCase`.
- **Без оверинжиниринга:** нет лишних слоёв «на вырост», своих DI-фреймворков и генерации кода. Type hints — по мере пользы, без жёсткого минимума в v1.
- **Зависимости:** только нужные пакеты в `pyproject.toml`; без optional-групп, пока нет реальной задачи.
- **Тесты в v1:** не обязательны; достаточно ручного прогона бота. Автотесты добавляем, когда появится регрессия или сложность.
- **Стили и линтеры в v1:** не закрепляем как обязательные (при желании позже можно подключить ruff и т.п.).

## 3. Структура проекта

Корень репозитория (минимум директорий и файлов):

```
project-root/
  pyproject.toml          # зависимости для uv
  uv.lock
  Makefile
  Dockerfile
  .env.example            # пример переменных окружения (без секретов)
  README.md               # запуск (по мере появления кода)
  docs/
    idea.md
    vision.md
  src/
    telegram_llm_bot/
      __init__.py
      main.py             # точка входа: Bot, Dispatcher, polling
      config.py           # загрузка настроек из окружения (класс настроек — один на файл)
      llm_client.py       # класс-обёртка над OpenAI-клиентом (OpenRouter)
      handlers.py         # хендлеры aiogram: допускаются функции в одном модуле
```

**Соглашение:** правило «один класс — один файл» относится к **классам**. Обработчики Telegram в v1 — **функции** в `handlers.py` (типовой стиль aiogram 3), без искусственного разбиения по одной функции на файл.

## 4. Архитектура проекта

**Один процесс, v1:** бот крутится до остановки, входящие сообщения обрабатываются последовательно в рамках выбранных хендлеров (без собственной очереди задач).

**Поток обработки:**

1. **Старт** — `main.py` загружает конфиг из окружения, создаёт `Bot` и `Dispatcher`.
2. **Регистрация** — к диспетчеру подключается роутер из `handlers.py` (личные текстовые сообщения).
3. **Сообщение пользователя** — хендлер передаёт текст в класс из `llm_client.py` вместе с системным промптом из конфига.
4. **Ответ** — готовый текст от модели отправляется пользователю **одним сообщением** (без стриминга в Telegram в v1).
5. **Цикл** — long polling aiogram до завершения процесса.

**Компоненты:**

| Компонент | Роль |
|-----------|------|
| `config` | Настройки из env: токен бота, ключ OpenRouter, идентификатор модели, системный промпт, при необходимости `base_url` |
| `llm_client` | Запрос к API чат-дополнений через пакет `openai` с `base_url` OpenRouter |
| `handlers` | Апдейты Telegram → вызов LLM → ответ пользователю |
| `main` | Сборка приложения и `run_polling` |

**Вне объёма v1:** базы данных, брокеры очередей, webhook, отдельный HTTP-сервис, кеш, мультиарендность.

## 5. Модель данных

- **Постоянное хранилище в v1 отсутствует** — без БД и без файлов с историей переписки.
- **Контекст для LLM:** системный промпт из конфигурации + **текущее** текстовое сообщение пользователя. История диалога **не** передаётся; каждый запрос к модели не опирается на прошлые реплики в чате.
- **Данные в коде:** настройки (класс в `config.py`) и сообщения роли `system` / `user` при вызове API; отдельной схемы сущностей нет.

**Позже (по необходимости):** можно добавить хранение последних N сообщений **в памяти процесса** по `user_id` — без БД и без изменения общей архитектуры v1.

## 6. Работа с LLM

- **Клиент:** пакет `openai` (`OpenAI`), совместимый с OpenRouter.
- **`base_url`:** по умолчанию `https://openrouter.ai/api/v1`; при необходимости переопределяется переменной окружения (см. раздел про конфигурацию).
- **Ключ:** `OPENROUTER_API_KEY` — передаётся в клиент как `api_key`.
- **Модель:** `OPENROUTER_MODEL` — строка в формате OpenRouter (конкретное значение не зашиваем в репозиторий).
- **Запрос:** для каждого сообщения пользователя один вызов `chat.completions.create` с массивом сообщений: роль `system` (промпт из конфига) и роль `user` (текст из Telegram).
- **Параметры генерации:** опционально `LLM_TEMPERATURE` и `LLM_MAX_TOKENS`; если не заданы — разумные константы по умолчанию в коде.
- **Ошибки провайдера:** пользователю — короткое нейтральное сообщение (например, что сервис временно недоступен); детали — в лог (раздел про логирование).

**Вне v1:** стриминг ответа, tool calling, выбор модели из чата, несколько провайдеров.

## 7. Сценарии работы

**Основной поток**

1. Пользователь пишет боту **текст** в **личном** чате.
2. Бот запрашивает ответ у LLM и отправляет **одно текстовое сообщение** с результатом.
3. История переписки не участвует в запросе (см. модель данных).

**Команды v1**

- **`/start`** — краткое приветствие: что это за JTBD-консультант, какие темы охватывает и что можно писать обычными сообщениями.
- **`/help`** (опционально) — та же суть, что и у `/start`, без отдельного сложного меню.

**Личный чат только**

- Сообщения **не** из лички (`private`) в v1 **не обрабатываем** — не отвечаем и не логируем как ошибку (минимум шума в группах).

**Не текст**

- Стикеры, фото, голос и т.п. — один короткий ответ: попросить отправить **текст** (формулировка на русском, без детализации типов медиа).

**Сбои**

- Ошибка LLM или сети — пользователю нейтральное сообщение (как в разделе про LLM); автоматических повторных запросов в v1 нет — пользователь может отправить сообщение снова.

## 8. Подход к конфигурированию

- **Источник настроек:** переменные окружения. Для локального запуска файл **`.env`** в корне репозитория подхватывается через зависимость **`python-dotenv`** при старте (если файл есть); значения из окружения имеют приоритет над `.env` там, где это принято для `load_dotenv`.
- **`config.py`:** загрузка и разбор env, проверка обязательных полей на старте; при отсутствии или пустом значении — ясное сообщение в лог и завершение процесса с ненулевым кодом.
- **Секреты** не храним в коде и не коммитим; в репозитории только **`.env.example`** с перечислением ключей без реальных значений и с короткими комментариями.

**Обязательные переменные (v1)**

| Переменная | Назначение |
|------------|------------|
| `TELEGRAM_BOT_TOKEN` | Токен бота (BotFather) |
| `OPENROUTER_API_KEY` | Ключ OpenRouter |
| `OPENROUTER_MODEL` | Идентификатор модели у провайдера |
| `SYSTEM_PROMPT` | Системный промпт (роль JTBD-консультанта; задаётся в env, не хардкодится) |

**Опциональные**

| Переменная | Назначение |
|------------|------------|
| `OPENROUTER_BASE_URL` | База API; по умолчанию `https://openrouter.ai/api/v1` |
| `LLM_TEMPERATURE` | Температура; иначе значение по умолчанию в коде |
| `LLM_MAX_TOKENS` | Лимит токенов ответа; иначе значение по умолчанию в коде |
| `LOG_LEVEL` | Уровень логирования (см. следующий раздел) |

---

## 9. Подход к логированию

- **Библиотека:** стандартный модуль **`logging`**, без сторонних сборщиков логов.
- **Настройка при старте:** базовая конфигурация (уровень, формат с уровнем и именем логгера, поток вывода — **stderr**). Уровень задаётся переменной **`LOG_LEVEL`** (`INFO`, `DEBUG` и т.д.); если не задан — **`INFO`**.
- **События:** старт и корректная остановка polling; ошибки вызова LLM и необработанные исключения — с текстом исключения и при необходимости кодом/статусом ответа API, без вывода огромных тел ответов.
- **Не логируем:** токены и ключи (`TELEGRAM_BOT_TOKEN`, `OPENROUTER_API_KEY`); **текст сообщений пользователя** в v1 не пишем в лог (ни на каком уровне), чтобы не раздувать объём и не хранить лишнюю персональную информацию.

## 10. Сборка и деплой

**Без Docker (macOS, локально)**

- **`make install`** — установка зависимостей: `uv sync` по `pyproject.toml` / `uv.lock`.
- **`make run`** — запуск бота через `uv run` (модуль пакета `telegram_llm_bot`).

**С Docker (локально)**

- **`make docker-build`** — сборка образа из `Dockerfile` в корне репозитория (базовый образ Python 3.12, зависимости через `uv`).
- **`make docker-run`** — запуск контейнера с пробросом переменных окружения (`--env-file .env`).

**Деплой в Railway (облако)**

Railway запускает Docker-контейнер напрямую из репозитория или из собранного образа.

- **`railway.toml`** в корне репозитория: указывает команду запуска (`CMD`) и сборщик (`DOCKERFILE`); Railway подхватывает его автоматически.
- **Переменные окружения:** все секреты (`TELEGRAM_BOT_TOKEN`, `OPENROUTER_API_KEY` и т.д.) задаются **только** через Railway Dashboard → Variables; в репозитории их нет.
- **Long polling** работает без изменений: Railway не требует открытого порта для бота-поллера; переменная `PORT` Railway не используется.
- **Деплой:** `git push` в ветку → Railway автоматически пересобирает образ и перезапускает сервис.
- **Логи:** видны в Railway Dashboard → Logs в реальном времени.

**Что не делаем в v1:** webhook, привязка custom domain, несколько окружений (staging/prod), авто-масштабирование.
