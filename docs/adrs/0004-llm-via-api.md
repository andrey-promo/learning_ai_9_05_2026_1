# ADR 0004: Доступ к LLM через API провайдера

## Статус

Принято

## Дата

2026-05-09

## Контекст

По [описанию идеи](../idea.md) поведение ассистента задаётся **системным промптом**, ответы генерирует **большая языковая модель**. Нужна техническая граница: как код проекта подключается к модели.

## Решение

Интеграция с LLM выполняется через **программный интерфейс внешнего провайдера** (HTTP API или SDK поверх него), а не через встраивание весов модели в репозиторий.

**Конкретный поставщик модели (OpenAI, Anthropic, совместимый прокси, региональный хостинг и т.д.) в этом ADR не фиксируется** — чтобы не противоречить будущему выбору. Его следует зафиксировать отдельным ADR после согласования требований (стоимость, регион, лимиты, качество).

Обязательство уровня архитектуры: промптинг (включая системный промпт) и обработка ответов остаются в коде приложения; секреты доступа к API — только через конфигурацию окружения, без попадания в Git.

## Последствия

### Положительные

- Можно менять модель и провайдера без смены канала Telegram.
- Нет необходимости с первого дня обслуживать собственный GPU-кластер.

### Отрицательные / риски

- Зависимость от доступности и политики провайдера.
- Учёт стоимости токенов и защита от злоупотреблений на стороне бота.
